# Entry skill: YAML templates with embedded fill instructions

## Problem

The entry skill deliberately avoids knowing the structure of its artifacts (`research.md`, `draft.md`). Section headings, purpose text, and fill procedure all live in `templates/`, so the skill stays generic and the templates can evolve without skill changes.

In practice, filling a research artifact well requires more than a one-line purpose description per section. There are format rules ("bulleted, one source per bullet, with a link"), content rules ("phrase as actual questions, not topics"), process hints ("search adjacent terms while gathering sources"), and cross-cutting rules ("keep Sources separate from Observations — never paraphrase a source into an observation without citing"). These accrue over time as the author refines how they want research done.

Today there is nowhere for this procedural guidance to live:

- **Inline in the template** — purpose placeholders sit inside `[...]` and get replaced the moment a section is filled. Any instruction written alongside them disappears after first fill, so subsequent revises have nothing to consult.
- **In SKILL.md** — ties the skill to a specific template's sections, breaking the separation the skill was built to preserve.
- **In the filled entry file** — clutters a working artifact that should be data-only.

The instructions need a home that is persistent across revises, co-located with the structural definition they belong to, and invisible to the per-entry artifact.

## Solution

Replace the markdown template files with structured YAML templates that define sections, per-section placeholder text, per-section fill instructions, and a top-level `rules` list for cross-cutting guidance. The per-entry artifact is rendered from the YAML and contains only headings and placeholders — never instructions.

### File format

Every artifact named in the skill's Artifacts section has a corresponding `templates/<artifact>.yaml`. Example:

```yaml
# templates/research.yaml
rules:
  - Keep Sources separate from Observations — never paraphrase a source into an observation without citing.

sections:
  - heading: Rough Idea
    description: |
      The seed of the entry: what you want to explore and why it's worth exploring. This section anchors direction — everything downstream builds from it, so it should be specific enough that someone else could tell what the entry is reaching for.
    instructions: |
      Expand the user's rough idea only enough to clarify direction. Do not over-commit to a thesis before observations exist.

  - heading: Questions
    description: |
      The core inquiries driving the exploration — what you need to probe or resolve to turn the rough idea into a claim.
    instructions: |
      Phrase as actual questions, not topic labels. Expect them to evolve as observations accumulate.
```

### Schema

- `rules` (optional list of strings) — cross-cutting procedural guidance that applies across sections.
- `sections` (required list, non-empty) — ordered list of sections in the order they appear in the rendered artifact.
  - `heading` (required string) — the section heading, rendered as `## {heading}`.
  - `description` (required string, may be empty) — user-facing purpose prose, rendered in the entry file wrapped in `[...]`.
  - `instructions` (required string, may be empty) — how to fill or revise this section. Never rendered into the entry file.

### Skill contract

The contract applies to **rendered artifacts** — artifacts whose on-disk form is rendered from a template on create. It does not apply to artifacts generated by an action (see below).

1. **Required template.** For each rendered artifact, `templates/<artifact>.yaml` must exist. Missing or malformed templates are a user-facing error with a clear message.
2. **Create by rendering.** When creating a rendered artifact, parse the YAML and write the entry file as the concatenation, in order, of each section rendered as:

   ```
   ## {heading}

   [{description}]
   ```

   `rules` and `instructions` are not written to the entry file.
3. **Fill and revise by consulting.** When filling a section on creation, or when revising a rendered artifact, read the YAML and follow the matching section's `instructions` plus the top-level `rules`. Match sections by heading text. Never write instructions or rules into the entry file.

The existing behavior around first-fill creation ("populate only the section whose purpose the rough idea directly addresses; leave other placeholders intact") is preserved — the YAML's `description` field plays the role the current `[...]` placeholders play today.

**Rendered vs generated artifacts.** `research.md` is the only rendered artifact. `draft.md` is *generated* by the Move to Draft action, which reads `research.md` in full and writes `draft.md` from scratch with an H1 title and a body informed by the research. Generated artifacts have no template — their guidance lives in the action's specification in SKILL.md's Per-Action Behavior. This asymmetry is intentional: template rendering and action-driven generation are different operations, and forcing one contract over both would either clutter the entry (if draft rendered from a template) or create schema ballast (if draft had an empty template never actually used).

### What changes

- `templates/research.md` → `templates/research.yaml`. Existing purpose text moves into `description` fields. New `instructions` fields and optional `rules` list are added per the author's preferences.
- `templates/draft.md` is deleted without replacement. Draft guidance continues to live in SKILL.md's Move to Draft and Revise Draft actions.
- SKILL.md Artifacts section is updated: research references `templates/research.yaml`; draft references the Move to Draft action.
- SKILL.md adds the Template contract above in a new short subsection under Artifacts, explicitly scoped to rendered artifacts.
- Per-entry artifacts continue to be written to `entries/wip/<id>-<slug>/research.md` and `draft.md`. Only the research template changes format; draft is unaffected at the per-entry level.

### What does not change

- Workflow, state model, commands, per-action behavior, and user-facing UI are unchanged.
- Per-entry artifacts remain plain markdown with headings and bracketed purpose placeholders in unfilled sections.
- The separation between skill and template knowledge is preserved — the skill references the YAML generically; only the YAML knows what sections exist.

## Tradeoffs

- **YAML editing friction vs markdown.** Editing purpose prose happens inside YAML block scalars rather than a plain `.md` file. With `|` block scalars this is livable for a small number of sections, and the payoff — structure, procedure, and placeholder all evolving together in one file — outweighs the cost.
- **One more parse step.** The skill must parse YAML and render to markdown on create. Trivial, and justified by the contract it enables.
- **Template contract covers only rendered artifacts.** Generated artifacts (currently `draft.md`) have no template and carry their guidance in the action spec. Accepted because rendering and generation are genuinely different operations; forcing one contract would either put procedure into the entry file or create an unused template. The carve-out is named explicitly in the Artifacts section so the asymmetry is discoverable.
